# 2026-04-14 記憶紀錄

## OpenClaw 系統可靠性與 UI 同步增強 (Reliability Guard v3.1)

### 1. 架構實作
實作了四層防護架構以解決 API 429/500 錯誤導致的 UI 停滯問題：
- **`rate-limit-proxy` (Port 18799)**: 監控 Google API 錯誤 \right → 觸發刷新指令。
- **`sync_sidecar` (Port 18790 & 18793)**: 
  - `18790`: WebSocket Server，與 UI 保持長連接，推送 `FORCE_RELOAD` 指令。
  - `18793`: HTTP Server，接收來自 Proxy 的刷新觸發請求。
- **`injection_proxy` (Port 18792)**: 
  - 攔截 UI 請求並注入 WebSocket 客戶端 JS。
  - **(關鍵修復)**: 自動注入 `Authorization: Bearer <token>` 標頭，解決 `auth=token` 模式下的 401 Unauthorized 死鎖問題。
- **`openclaw-gateway` (Port 18791)**: 基礎 UI 渲染。

### 2. 核心機制演進
- **從 Polling 轉向 Push**: 將原本 10 秒一次的版本輪詢升級為 WebSocket 即時推送，將恢復延遲從秒級降低至毫秒級。
- **解決認證死鎖**: 發現 Gateway 在 `auth=token` 模式下會拒絕未帶 Token 的初始 HTML 請求，導致 UI 無法載入。透過在 `injection_proxy` 層強制注入 Token 解決。

### 3. 關鍵端口映射
- `18789`: Gateway API
- `18790`: Sidecar WebSocket (UI 連接)
- `18791`: Gateway UI (原版入口)
- `18792`: Injection Proxy (增強版入口 - 推薦)
- `18793`: Sidecar HTTP (指令接收)
- `18799`: Rate-Limit Proxy (流量守衛)

### 4. 部署狀態
- 所有組件均已透過 macOS `launchd` (LaunchAgents) 設定為 `KeepAlive`，實現故障自動重啟。
