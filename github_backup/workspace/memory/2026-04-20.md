## 2026-04-20 19:41 - 攔截憲法更新與備份失效分析

### 1. $\text{LaTeX}$ 污染修正
- **事件**：在回報備份進度時，輸出中出現 `\text{GB}` 與 `\le` 之原始代碼，導致 $\text{LaTeX}$ 污染。
- **根因**：由於符號未被 `$` 包圍，未觸發初步攔截，且 `intercept_rules.json` 中缺乏對 `\le` 與 `\text{GB}` 的具體映射。
- **修正**：已將 `\\le` $\rightarrow$ `≤` 與 `\\text{GB}` $\rightarrow$ `GB` 寫入 `intercept_rules.json` 憲法攔截表。

### 2. 全量靈魂備份 (Full Soul Snapshot) 失敗分析
- **狀態**：本地打包成功 (48 個分卷 $\approx 2.4\text{GB}$)，但雲端同步 $\rightarrow$ `git push` 階段失敗。
- **錯誤碼**：`MALFORMED_RESPONSE`。
- **深層分析**：
    - 物理層：單次巨量二進位數據 Push 導致傳輸阻塞 (Transport Blocking)。
    - 運行時：`Sub-agent` 在等待高延遲傳輸結果時，與 Provider 之間的心跳同步失效 $\rightarrow$ 觸發 Provider 端崩潰。
- **後續對策**：
    - 短期：執行「純同步補完」，直接推送本地已就緒的分卷。
    - 長期：將「單次巨量 Push」升級為「分批 Push」 (例如每 10 個分卷 commit/push 一次)，消除傳輸層超時風險。

## 2026-04-20 20:31 - 雲端同步策略升級方案：分批 Push (Batch Push)

### 1. 背景與痛點
- **問題**：在執行 $\approx 2.4\text{GB}$ 的全量靈魂備份時，即使採用了分卷方案 ($\le 50\text{MB}$)，但在同步至 GitHub 時，若將所有分卷在單一 Commit 中推送，會觸發傳輸層阻塞 (Transport Blocking) 與 Provider 端心跳超時 $\rightarrow$ 導致 `MALFORMED_RESPONSE` 崩潰。
- **結論**：單次巨量 Push 的魯棒性不足，無法應對極大體積的二進位數據同步。

### 2. 升級提案：分批 Push 策略 (Batch Push Strategy)
- **核心邏輯**：將「單次巨量 Push」 $\rightarrow$ 「多批次微量 Push」。
- **實作流程**：
    1. **分組**：將所有分卷（例如 48 個）分為多組（每組 5-10 個分卷）。
    2. **循環同步**：
       - `git add` 第一組分卷 $\rightarrow$ `git commit` $\rightarrow$ `git push`。
       - `git add` 第二組分卷 $\rightarrow$ `git commit` $\rightarrow$ `git push`。
       - $\dots$ 依此類推直到所有分卷上傳完畢。
    3. **最終驗證**：上傳所有分卷後，最後推送 `backup_manifest.json` 以鎖定整個備份集的完整性。
- **預期效果**：
    - **消除超時**：單次傳輸量大幅下降，避免觸發 GitHub 或 Provider 的超時機制。
    - **精確斷點**：若在第 3 組發生失敗，系統能明確知道是哪些分卷出錯，無需從頭開始。
    - **提高成功率**：將單一高風險任務拆分為多個低風險微任務。

### 3. 技能升級預備
- 此方案將作為 `openclaw-soul-sync` 技能之核心升級邏輯。
- 需在 `soul_sync_execute.py` 中引入 `batch_size` 參數，實現自動化分批提交邏輯。
