# 最終分析報告：資料重複輸出問題與特色食物抓取優化
**日期：** 2026-04-13 
**狀態：** 已結案 / 進入維護期 
**對應技能版本：** `where-to-go v2.25` | `google-places v10.6`

---

## 📋 執行概覽
本報告整合 2026-04-12 之多次研究結果，針對使用者反映之「資料重複輸出」及「食物抓取不精準」問題進行深度分析與修復。

### 核心問題清單
1. **雙重輸出（Duplication）**：輸出內容出現原文字與翻譯文並列（如：🚗 開車 / 🚗 駕車）。
2. **評論截斷（16行慘狀）**：長英文或無空格字串被錯誤截斷成單字一行。
3. **抓取量不足**：1km 內店家輸出數量低於預期（CDP 超時）。
4. **特色菜色缺失**：部分店家的特色食物未能正確識別或呈現。

---

## 🔍 一、資料重複與格式崩潰分析

### 1.1 「雙重輸出」之真相：Qclaw 翻譯層
**分析過程：**
- 隔離測試發現 `python3 run.py` 直接執行結果完全乾淨。
- 檢查 `openclaw.json` 無翻譯設定 \right → 定位至 `Qclaw.app` 應用程式層。
- **結論**：Qclaw 應用程式層對非正體中文內容觸發了「自動翻譯志願者」機制，將翻譯後結果與原文同時呈現在 UI 上。

**解決方案：**
- **移除冗餘翻譯**：從 `run.py` 中完全移除 `deep_translator` 邏輯，確保技能層不主動觸發翻譯。
- **防呆去重**：在渲染邏輯中加入 `if wline not in lines_out` 檢查，防止同一行被重複添加。
- **防翻譯標記**：將評論內容用反引號 (`` ` ``) 包裹，降低被系統翻譯層識別為「待翻譯文本」的機率。

### 1.2 「16行慘狀」之真相：Wrap 邏輯 Bug
**分析過程：**
- 發現 `_wrap_chars` 函數在處理長度 >50 且無空格的字串時，每次遞迴僅截取單個字元。
- **結論**：遞迴終止條件不足，導致長字串被強行拆分為單字行。

**解決方案：**
- **修正 `_wrap_chars`**：當剩餘文字長度 \le 指定寬度時，直接輸出剩餘全部文字，停止遞迴截斷。

## 🍲 二、特色食物抓取優化分析

### 2.1 現有抓取雙軌制
目前採取 **CDP 官方標籤** \times **評論正則分析** 的混合方案：

| 方法 | 技術路徑 | 優勢 | 局限 |
|------|----------|------|------|
| **CDP 分類** | 抓取 `[role=navigation]` 按鈕 | 官方定義、速度快 | 僅為大類（如：咖啡），非具體菜色 |
| **正則分析** | `_DISH_PATTERNS` \right → 評論文字 | 具體、含評論片段 | 依賴評論數量與關鍵字覆蓋率 |

### 2.2 關鍵優化實施
- **模式庫大擴充**：`_DISH_PATTERNS` 從 59 條擴充至 **282 條**，全面覆蓋台灣早餐店所有品項（蛋餅、飯糰、豆漿、蘿蔔糕、碗粿、鐵板麵等）。
- **CDP 超時修復**：
  - `click_tab()` 等待時間 3s \right → 1s。
  - Phase 3 價格彈出等待 4s \right → 2s。
  - 修正 Chrome 預熱 PORT 為 `18800`，減少啟動延遲。
- **評論合併邏輯 (v2.25)**：同一作者且時間相近的評論片段自動合併，解決評論被切割成碎片的問題。

## 🚀 三、未來路線圖 (Roadmap)

雖然目前問題已解決，但為達到「極致精準」，計劃於後續版本實施：

1. **整合 Google Places API (New) `dishes` 欄位**：直接獲取官方結構化菜單，不再依賴爬蟲。
2. **引入 `reviewSummary` 欄位**：利用 Google AI 生成的評論摘要，直接提取熱門菜色。
3. **放寬 CDP 分類長度限制**：將 `t.length <= 10` 提高，以捕捉更長的分類名稱（如「傳統台式早餐」）。
