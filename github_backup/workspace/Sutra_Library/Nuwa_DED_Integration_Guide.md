女媧系統與虛擬專家工作小組 (DED) 整合使用指南
本指南旨在詳細說明 女媧 (huashu-nuwa) 技能與 虛擬專家工作小組 (DED - Dynamic Expert Dispatcher) 之間的協作邏輯。這套系統將「認知提煉」與「即時調度」相結合，讓 AI 能在不同專家視角之間快速切換，提供高純度的專業分析。

第一章：虛擬專家工作小組 (DED) 核心架構
1.1 系統定義
「虛擬專家工作小組」並非單一程式，而是一套由三個核心組件構成的協作機制：

nuwa-dispatcher (調度員)：扮演「大腦」角色，負責意圖識別與認知分流。
EXPERT_REGISTRY.md (能力圖譜)：扮演「索引」角色，記錄所有可用專家的認知 DNA。
huashu-nuwa (執行引擎)：扮演「身體」角色，負責根據注入的 DNA 執行具體的分析輸出。
只要這三個組件配置完成，系統即處於隨時待命狀態。

1.2 即時認知分流流程
當使用者提問時，系統會啟動**「即時認知分流」**模式：

意圖偵測：調度員掃描輸入，識別觸發詞（如：「專家的看法」、「專家分析」等）。
即時匹配：調度員調用 scripts/dispatch.py 掃描 EXPERT_REGISTRY.md，根據問題領域與思考風格，毫秒級篩選出最合適的「專家原型」。
DNA 注入：將選定專家的「認知 DNA」（思考風格、邏輯模式）即時注入 huashu-nuwa 引擎，使其轉化為該專家身份進行回答。
第二章：女媧 (huashu-nuwa) 的專家蒸餾機制
2.1 專家技能的存放路徑
女媧系統將專家視為一套可運行的認知操作系統，其存放路徑分為兩類：

內建範例專家：
路徑：/Users/KS/.openclaw/workspace/.agents/skills/huashu-nuwa/examples/[專家名稱]-perspective/
(例如：Andrej Karpathy 的路徑為 .../examples/andrej-karpathy-perspective/)
新蒸餾專家：
路徑：.claude/skills/[專家名稱]-perspective/
設計核心：追求**「自包含 (Self-contained)」**。每個專家目錄不僅包含 SKILL.md 認知框架，還包含所有调研證據 (references/)，確保技能可獨立複製分發，無需依賴外部數據庫。

2.2 從蒸餾到固化的過程
女媧在「造人」過程中採取物理與邏輯雙軌制：

物理存儲 (認知固化)：通過生成詳細的 SKILL.md 文件，將心智模型、表達 DNA 與誠實邊界永久化。
邏輯登記 (發現機制)：將專家的基本定義（名稱、專長、原型、模式）登記至全局索引文件 EXPERT_REGISTRY.md 中。
第三章：系統聯動與溝通模式
3.1 聯動關係：大腦 → 索引 → 身體
女媧系統與 DED 之間形成了一種強依賴的共生關係：
nuwa-dispatcher (調度員/大腦) → EXPERT_REGISTRY.md (能力圖譜/索引) → huashu-nuwa (執行引擎/身體)

3.2 溝通模式：基於登記冊的認知注入 (Registry-based Cognitive Injection)
此溝通模式確保了調度的高效與輸出的純度：

請求接收 → nuwa-dispatcher 接收使用者對專家的需求。
特徵匹配 → 調度員在 EXPERT_REGISTRY.md 中尋找匹配條目。
DNA 提取 → 提取特定認知 DNA（例如：第一原理原型 → 物理拆解邏輯）。
身份注入 → 將 DNA 作為高權重指令注入 huashu-nuwa。
高純度輸出 → huashu-nuwa 切換推理模式，生成專家級分析。
第四章：維護與擴展指南
4.1 新專家的加入流程
當您希望增加一名新專家時，請區分以下兩種操作：

操作對象 執行動作 生效機制
增加「專家原型」 將定義寫入 EXPERT_REGISTRY.md 表格 立即生效 (調度員即時讀取)
增加「系統技能」 透過 skill-creator 框架安裝 .md 技能文件 重啟生效 (Gateway 掃描目錄)
4.2 重要警示：登記機制
huashu-nuwa 蒸餾出新專家後，不會自動登記至全局索引文件。

由於女媧主流程專注於「認知提煉 (造人)」，而 EXPERT_REGISTRY.md 屬於「調度層索引 (用人)」，兩者職能分離。
必要操作：在女媧交付新專家後，使用者必須將其基本定義手動或指令式地添加到 /Users/KS/.openclaw/workspace/EXPERT_REGISTRY.md 中，否則調度員將無法辨識並激活該專家。

4.3 辨識路徑依賴
對於「虛擬專家工作小組」而言，路徑並不影響辨識。系統是以 EXPERT_REGISTRY.md 裡的數據條目為準，只要登記在案，無論執行引擎安裝在何處，調度員均能通過名稱正確指派任務。

2. 專家持續升級機制
針對既有專家的持續升級，採取以下詳細執行路徑：

** (1) 是否可以直接針對既有專家進行持續升級？**
是的，完全可以。執行路徑分為：

路徑 A：定向增量更新 (Incremental Update)
如果您有新的素材（例如：該專家最近的一場訪談、一本新書或一批內部文件），您可以直接餵給女媧，並指示：「更新 [專家名] 的 Skill，加入這些新素材」。女媧會啟動特定的更新流程，將新資訊整合進現有框架。
路徑 B：全量重新蒸餾 (Full Re-distillation)
如果您認為目前的專家模型過時，或者想在更高的維度（例如從 2024 年的認知更新到 2026 年）重新定義，您可以要求「重新蒸餾 [專家名]」。這會讓女媧重新跑一遍完整的 Phase 0 $\rightarrow$ Phase 5 流程。
** (2) 原生支援度與損害評估**

原生支援度：這是在 SKILL.md 中明確定義的標準功能。更新協議包含：
時間戳檢查：讀取現有 SKILL.md 的「誠實邊界」章節，確認上次调研的時間點，以決定需要追溯多少新資訊。
精準Agent調用：優先啟動 Agent 2 (最新對話) + Agent 5 (最新決策) + Agent 6 (時間線更新) 捕捉變量。
對比與整合：判定為 $\text{強化}$ (補充案例) $\rightarrow$ $\text{矛盾}$ (標記認知轉向) $\rightarrow$ $\text{新增}$ (提取新模型)。
損害評估 (低風險)：採取「增量優先」保護機制。
禁令：禁止全量重寫，僅採取「追加」與「修正」方式，確保深度心智模型不被隨意刪除。
保留矛盾：將衝突記錄為「認知演化」或「內在張力」，使模型更立體。
** (3) 專家總結建議**
建議採取 「定向餵料 $\rightarrow$ 增量更新」 方式。
推薦指令示例：「女媧，我現在提供 [專家名] 最近三個月的所有播客轉錄稿，請針對其『表達 DNA』和『最新決策模式』進行增量更新，並同步更新 EXPERT_REGISTRY.md 中的認知 DNA。」
