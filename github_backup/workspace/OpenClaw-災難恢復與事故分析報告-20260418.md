# OpenClaw 事故分析與災難恢復技術報告 (2026-04-18)

## 1. 事故概述 (Incident Overview)
- **錯誤訊息**：`disconnected (1006): no reason`
- **發生時間**：2026-04-18 22:17:06
- **事故等級**：中高級 (Core Gateway Service Failure)
- **最終結果**：透過 v10.0 深度復甦計畫，成功恢復網關運作與區網通訊。

---

## 2. 故障根源解析 (Root Cause Analysis)
本次事故並非由網路不穩引起，而是由一次**自動化技能衝突**所觸發：
1. **觸發源**：系統自動執行了 `system-clone-local` (克隆管理技能)，該技能發出了 `pkill -f openclaw-gateway` 的指令。
2. **連鎖反應**：網關進程被強制結束，但由於該進程受到我們設置的 `uchg` (物理鎖定) 保管，macOS 的 `launchctl` 子系統在嘗試自動重啟時與物理鎖定發生衝突。
3. **系統死結**：`launchd` 由於無法正確更新服務標記，將 `ai.openclaw.gateway` 列入黑名單，產生了 `Input/output error` (5) 錯誤，導致標準恢復指令失效。
4. **外部表現**：核心網關崩潰，但外圍代理 (Proxy) 仍在運行，導致使用者在瀏覽器看到不斷重連的 `1006` 錯誤。

---

## 3. 恢復與修復流程 (Resolution Process)
為了解開系統層級的死結，我們執行了以下「身分繞道」修復：

### 步驟 A：解除物理鎖定
使用 `chflags nouchg` 暫時移除了 Plist 檔案的保護膜，讓 `launchctl` 有機會重新讀取檔案。

### 步驟 B：服務身分切換 (Critical Step)
由於原有的服務標籤 `ai.openclaw.gateway` 已被系統凍結，我們執行了以下操作：
- 將 Plist 內的 `<key>Label</key>` 修改為 **`ai.openclaw.gateway.v2`**。
- 這讓 macOS 認為這是一個全新、健康的服務，進而成功繞過了系統黑名單。

### 步驟 C：全系統重啟 (Sequential Restart)
按照「核心 -> 轉發 -> 加密」的黃金順序，強行恢復了 18792 埠口的監聽。

### 步驟 D：物理鎖定回歸
服務穩定後，再次施加 `chflags uchg`，防止未來的自動化任務自行竄改配置。

---

## 4. 未來預防與建議 (Preventive Measures)
1. **優化克隆技能**：建議將 `pkill` 指令從自動化備份腳本中移除，改用 `launchctl kickstart` 等更溫和的系統級管理指令。
2. **服務標籤演進**：目前的 `v2` 標籤已生效。若未來再次發生同類死結，可依照此邏輯切換為 `v3` 進行強制註冊。
3. **備援監控**：我已將此復甦邏輯整合進 Antigravity 核心技能中，下次發生類似狀況時，系統將具備「自我識別並繞道恢復」的能力。
