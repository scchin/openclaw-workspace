# 我要去哪（where-to-go）技能

---

## ⚠️ 最高優先原則（強制遵守，無例外）

### 天條（2026-03-23 新增）

> 本技能一次只做一件事：**完整執行，完整輸出。**
> 以下每一條都是高壓線，碰都別碰。

#### ０．核心行為準則（最高優先，2026-03-23 新增）
**拿到的完整輸出，就是回覆的完整內容。**
不多一個字，不少一個字。不摘要、不濃縮、不重新組織、不用自己的話說。

**每一輪對話的執行流程：**
1. 使用者說「我要吃 XX」→ 立即執行 `run.py`
2. `run.py` 回什麼，就把那段文字**原封不動**貼進回覆
3. 可以在「最後一行」加結尾語（「要吃哪間？」），但**不得修改、濃縮、或省略上面的原始輸出**

**為什麼要這樣規定：**
`run.py` 的輸出已經過格式化（12 欄位、完整店家資訊）。濃縮是對這個流程的不信任。
「使用者會覺得這樣比較好讀」是**假設**，不是用戶說的。用戶說的是「絕對不準省略」。

#### １．不幫用戶決定顯示什麼

#### ２．12 欄位 100% 完整輸出
**每一個店家區塊都必須包含全部 12 欄位，無例外。**
就算 CDP 抓到 `price=none`、`found=False`、或任何空值，**仍須顯示該欄位**，不得置換成 "店家未提供" 或任何替代文字。

#### ３．每人消費六層 fallback（與 CDP 原則互補，2026-04-01 更新）
原則１（CDP 優先）與原則２（完整呈現）之下，每人消費的實際取值順序如下：
- ① **Google Places API `priceRange`**（最準，Google 官方）
- ② **CDP 分布圖**（`maps_price.found=True` 且有 `price_range`）
- ③ **CDP fallback**（`maps_price.price_range` 存在但 `found=False`）
- ④ **評論價格正則解析**（從6個月內評論中抽 `$數字`）
- ⑤ **`price_level` 符號轉換**（`goplaces` API：`1=$ | 2=$$ | 3=$$$ | 4=$$$$`）
- ⑥ **`place_data.priceRange`**（Google API 的原始欄位，CDP 抓不到時的最後防線）

→ 當①–⑥全部失敗，才顯示「無」

**⚠️ 不得跳過上述順序自行決定顯示邏輯。**

#### ４．CDP 爬蟲必須強制執行（2026-03-23 新增）
**⚠️ 這是最高優先修正。**
- `run.py` 的 `search_and_detail` 函式在取得候選店家清單後，**必須對每一間店家執行 `query_place_full(place_id)`**
- `query_place_full` 內部會啟動 CDP 瀏覽器、抓取每人消費、服務、特色菜色摘要（popular items）、分布圖
- **不得**只靠 `goplaces search` 或 `query_place_api_only` 的基本欄位就結束
- **不得**在未實際呼叫 CDP 的情況下聲稱「已完整查詢」

#### ５．Progressive 時間窗口取評論（2026-03-23 新增）
評論的時間窗口依序擴展：**6 個月 → 1 年 → 2 年 → 全部**，
直到取到「最接近目前時間的 5 則」為止。
- 若 6 個月內有 ≥ 5 則，取最新的 5 則
- 若不足，擴展到 1 年；再不足，擴展到 2 年；再不足，取全部
- 天條：**只要 API 回的任何評論，就要設法取出並顯示特色菜色和網友心得**
- **不得**只因為「6 個月內沒有評論」就讓這兩個欄位顯示「無」

#### ６．不得隱藏任何作者的評論（2026-03-23 新增）
不得以作者身份（例如作者名稱）做黑名單過濾。
只要是 API 回的評論，無論作者是誰，都要納入特色菜色和網友心得的候選範圍。

#### ７．每人消費抓取放寬（2026-03-23 新增）
CDP 爬蟲的 JS_BASIC 價格 regex：
- **舊版**：要求「每人/per person」文字 + 價格範圍同時存在 → 許多店只秀 `$60~$170` 抓不到
- **新版**：只要價格範圍存在（`$數字~$數字` 或 `$數字-$數字`），且金額合理（最高 < 100,000），就視為有效
- 不強制要求「每人」文字出現在 DOM 中

#### ８．CDP 瀏覽器自動登入（2026-03-24 更新）
Google Maps 需登入才能看到完整評論與每人消費。
CDP 瀏覽器啟動時自動從 `~/Library/Application Support/Google/Chrome/Default` 複製已登入 profile 至 `/tmp/openclaw-chrome-cdp/`，確保 session 處於登入狀態。
此為 Google Maps 本身的限制，技能應如實呈現，**不得因此放棄抓取或竄改內容**。

#### ９．特色菜色關鍵字庫已擴充（2026-03-23 新增）
`DISH_ITEMS`（`query.py`）和 `_DISH_PATTERNS`（`run.py`）已加入早餐常見品項：
鐵板麵、蘑菇鐵板麵、陽春麵、蛋餅、蛋餅皮、鍋貼、水煎包、蔥油餅、餡餅、
蘿蔔糕、菜頭粿、碗粿、肉粽、飯糰、蚵仔麵線、麵線、豆漿、米漿、鹹豆漿、
甜豆漿、燒餅、油條、小籠湯包、水餃、餛飩麵、餛飩湯、肉圓、蚵仔煎、蚵仔、
肉燥飯、滷肉飯、雞腿飯、便當、自助餐、素食、蔬食等 40+ 種。
未來新增食物類型時，必須同步擴充此清單。

#### １．評論時間戳解析（2026-04-01 v2.17 新增）
Google API 的 publish_time 含 9 位小數（奈秒），Python 3.9 fromisoformat 僅支援 6 位。
未截斷會觸發 ValueError，導致所有評論顯示「今天」。技能使用 _parse_pub_time() 統一截斷至 6 位小數。

#### ２．地址格式 SWAP 3 規則（2026-04-01 v2.17 新增）
針對「河南路二段106號」這類門牌號黏在路段後方的格式，SWAP 3 以貪心正則 `(.*?)(\\d+號)$` 拆分出路名與門牌號。

#### ３．店名一律顯示繁體中文（2026-04-07 新增）
店家名稱**只准顯示 Google 官方繁體中文名稱**（由 `get_localized_name_addr()` 的 `name` 欄位提供）。
嚴禁出現：
- ❌ 英文店名（如 `Chang Ka Breakfast Restaurant`）
- ❌ 中英混雜（如 `Chuantong Breakfast Restaurant`）
實作方式：`format_full_output()` 從 `query_place_full()` 的 raw output 解析 `🏪 店名：中文名` 這一行，若有中文名則覆寫 `place_data.get("name")`。

#### ４．午餐搜尋排除早午餐店家（2026-04-01 v2.17 新增）
搜尋「午餐」（非早午餐/早餐）時，Google Maps 會混回大量 brunch_restaurant。
處理方式：早午餐店家（type 含 brunch_restaurant 或名稱含「早午餐」/brunch）降序排到列表末端。
非早午餐 ≥ 2 家時只取非早午餐；否則兩者皆保留以確保結果數量。
is_brunchy() 統一判斷；goplaces_search 傳入 max(5,max_results) 確保過濾後仍有足夠候選。

#### ５．特色菜色 pattern 大幅擴充（2026-04-01 v2.17 新增）
_DISH_PATTERNS 已從 40+ 擴充至 200+ 條目，涵蓋：
- 「形容詞+蛋白質」組合（香草雞、胡麻豬、明太子貝等）
- 精確品項（霹靂霹靂牛、胡麻豬、剝皮辣椒雞等）
特色顯示使用 dict.fromkeys() 去重並保持順序（最多5項）。

#### ６．💵每人消費 六層 fallback（2026-04-01 v2.17 新增）
① Google Places API priceRange（最準）→ ② CDP 有找到 → ③ CDP fallback → ④ 評論價格正則解析 → ⑤ goplaces price_level 兜底
當所有上層都失敗時，用 price_level（1=$, 2=$$, 3=$$$, 4=$$$$）轉換為 $ 符號。

#### ７．🗺️地圖連結修正（2026-04-01 v2.17 新增）
maps_url 的 place_id 直接取 place_data.get("place_id")，不依賴 api_data 傳入。

#### ８．filter_and_sort 擴張 break 條件（2026-04-01 v2.17 新增）
半徑擴張迴圈 break 條件：根據「非早午餐店家數量」判斷是否繼續擴張，
而非根據「總候選數量」（總數可能在第一圈就滿足但過濾後不足）。

## 完整流程

```
Step 1 → 解析使用者輸入（關鍵字、圓心、半徑）
Step 2 → 半徑遞增搜尋候選地點（累積去重，持續擴張直到滿足數量）
Step 3 → 過濾 + 排序（評分 ≥ 門檻，由近到遠）
Step 4 → 【食物關鍵字】評論過濾（Progressive 6M→1Y→2Y→全部）
Step 5 → 取評分最高的目標數量店家作為候選
Step 6 → 重新按距離由近到遠排列（★★★ 2026-03-22 新增）
Step 7 → 對每間店家執行 CDP 爬蟲（query_place_full），抓取每人消費、特色菜色摘要、服務、評論
Step 8 → 依距離順序輸出所有結果
```

---

## Step 1｜解析使用者輸入

檢查使用者是否提供以下資訊：

| 資訊 | 說明 |
|------|------|
| 所在地 | 地址、地點名稱、經緯度 |
| 範圍半徑 | 公尺數、開車分鐘數、或走路分鐘數 |
| 數量 | `--count N`，預設 5 家 |

**半徑換算**：
- `500m` 或 `500` → 500 公尺
- `1.5km` → 1500 公尺
- `開車 10 分` → 500 × 10 = 5000 公尺
- `走路 15 分` → 80 × 15 = 1200 公尺

**預設值**（使用者未提供任何地點時）：
- 圓心：`24.18588146738243, 120.66765935832942`（我家）
- 半徑：`1000m`
- 數量：**5 家**（2026-03-26 確認）

---

## Step 2｜半徑遞增搜尋候選地點（★★★ 2026-03-26 修正）

**⚠️ 2026-03-26 重大修正：半徑自動持續擴張，直到湊足數量。**

```bash
goplaces search "<關鍵字>" \
  --lat <緯度> --lng <經度> --radius-m <半徑> \
  --open-now --min-rating 4.5 --limit 20 --json
```

**擴張順序**（每次自動遞增）：
```
1000 → 1500 → 2500 → 5000 → 10000 → 20000 → 50000 m
```

**累積去重邏輯（2026-03-26 修正）：**
- 每次半徑擴張取得的候選，與前次結果**合併**（依 `place_id` 去重）
- 不再是覆蓋模式，確保擴張半徑時已找到的店家不會消失
- 持續擴張直到候選數量 ≥ 目標數量（預設 5 家）

**預設參數**：
- `--open-now`：只看現在有開的
- `--min-rating 4.5`：評分 4.5 以上
- `--limit 20`：goplaces 每次最多取 20 筆

---

## Step 3｜過濾 + 排序

1. 移除 `CLOSED_PERMANENTLY`（永久停業）
2. 評分低於門檻 → 移除
3. **由近到遠**排序（Haversine 直線距離）
4. 取前 **5 名**（2026-03-26 確認）

---

## Step 4｜食物關鍵字過濾（★★★ 2026-03-23 更新）

當使用者輸入包含**食物關鍵字**（如「火鍋」「素食」「港式」）時：

1. 對每個候選地點呼叫 **Google Places API** 取得評論
2. 掃描範圍：
   - 店家名稱、類型、說明
   - 熱門品項（popular items）
   - **評論原文**（原文優於翻譯）
3. 比對方式：
   - 直接關鍵字（火鍋 → `火鍋`、`涮涮鍋`、`麻辣鍋`、`鍋物`）
   - 別名擴充（如「港式」包含`茶餐廳`、`燒臘`、`腸粉`、`叉燒`）
4. **只有提到該食物的店家才入選**，其餘替補至數量補足
5. 若替補後仍不足 5 家，放寬至全部過濾後結果

## Step 5｜CDP 爬蟲完整查詢（★★★ 2026-03-24 更新）

**⚠️ 強制執行：每一個入選地點都必須執行 CDP 爬蟲，直接輸出完整內容，不跳過任何欄位。**

`run.py` 的 `search_and_detail` 函式在遍歷候選店家時，**對每一間**呼叫 `query_place_full(place_id)`：
- 啟動 CDP 瀏覽器（Edge → Chrome fallback）
- 自動複製 Chrome 已登入 profile（`~/Library/Application Support/Google/Chrome/Default` → `/tmp/openclaw-chrome-cdp/`），確保 CDP session 處於登入狀態
- Phase 2：抓取「每人消費」（JS_BASIC regex 價格範圍）
- **Phase 3（無條件執行）：** 無論 Phase 2 是否找到價格，都去頁面下方抓「平均每人」+ 分布圖
- 抓取「服務」標籤（內用/外帶/外送/寵物友善等）
- 抓取「特色菜色摘要」（popular items）
- 抓取「價格分布圖」（price_histogram）
- 同時取 API 評論（Progressive time window：6M→1Y→2Y→全部，最多 5 則）

**執行方式（由 `run.py` 內部呼叫，不手動執行）：**
```python
structured, raw_output = query_place_full(place_id)
```

**⚠️ 嚴禁事項：**
- 不得跳過 CDP 直接用 `goplaces search` 基本欄位當作完整輸出
- 不得只靠 `query_place_api_only` 就聲稱已完成查詢
- 不得在輸出前詢問用戶「要不要查詳細」，預設就是全部查完輸出

## Step 6｜最終輸出

## 注意事項

1. **食物關鍵字智慧比對**：Google 評論中只有明確提到相關食物的店家才入選，並支援常見別名（如「素食」含「蔬食」、「Vegan」）
2. **當下有開**：`--open-now` 過濾，並在輸出中標注營業狀態
3. **評分門檻**：預設 4.5，使用者可調整（如「4.0 分以上也可以」）
4. **數量預設**：5 家；使用者明確要求 N 間時動態調整為 N
5. **半徑擴張**：最多擴張到 50km，以找到足夠店家為主；採累積去重模式
6. **完整查詢費時**：5 家約 1–2 分鐘，10 家約 2–4 分鐘（Google Places API + CDP），已自動執行不需詢問
7. **電話 / 營業時間**：goplaces search 結果中電話不一定有，完整查詢會補足
