## 二、問題重現失敗

嘗試多種方式，均**無法重現**用戶描述的：
- 「🚗 開車：1分｜🚶 走路：3分（0.2km）」 + 「🚗 駕車：1 分｜🚶 步行：3 分（0.2 公里）」同時出現
- 「👤 仙草：最初內用...」 + 「👤 仙草：最初是內用客戶...」同時出現

**結論：問題可能不在技能代碼本身。**

## 三、可能原因（依可能性排序）

### 可能性 1：OpenClaw WebChat 渲染層問題（最可能）

症狀描述與「雙重渲染」吻合：
- 技能輸出（原始） + 翻譯層輸出（翻譯）→ 兩套內容同時出現
- 這可能發生在 OpenClaw 的 `webchat` channel 的某個中介層

**驗證方式：**
```bash
# 查看 OpenClaw webchat 的原始輸出（需 access log）
```

### 可能性 2：緩存導致的新舊代碼混合

若系統有兩份 `run.py`（備份 + 活躍）且加載順序不穩定：
- 舊版：含翻譯邏輯
- 新版：已移除翻譯
- 但某些 session 仍載入舊版

**驗證方式：**
```bash
# 確認所有 `.py` 檔的 mtime
ls -la ~/.openclaw/skills/where-to-go/scripts/
md5 run.py
```

### 可能性 3：上一次正常輸出發生在 v2.24，之後的更新引入了某個隱藏的迴歸

需比較 v2.24 與 v2.26 的 `run.py` 差異（特別是 `format_full_output`、`_rev_from_list`、`query_place_full` 函式）。

## 四、改善方案

### 方案 A：立即可行（預防性）

**在 `format_full_output()` 中加入防呆機制：**
```python
# 確保同一則評論只有一個呈現（去重）
shown_notes = set()
# 在渲染評論前先做文字去重
```

**在 `_rev_from_list()` 確認無重複評論：**
```python
# 增強 seen_text 比對（不只比對作者+天數，而是完整內容比對）
```

### 方案 B：中長期（根本解決）

1. **完全移除 `deep_translator` 翻譯邏輯**
   - 改為直接使用 `original_text`（評論原文）
   - 這符合「數據真實性原則」
   - 預防翻譯干擾

2. **統一 `REV_COUNT` 為 10（每店最多 10 則）**
   - 避免少數店家評論過少時的邏輯分支錯誤

3. **單一評論渲染管道**
   - 廢除價格評論（`🍴 💰 👤`）的單獨渲染
   - 統一走 `_rev_from_list()` → `format_full_output()` 線性流程

4. **加載日誌機制**
   - 在每次 `query_place_full()` 和 `format_full_output()` 入口寫入時間戳
   - 協助追蹤是否有双重執行

### 方案 C：系統層面（需 OpenClaw 團隊確認）

檢查 OpenClaw 的 `webchat` channel 是否有：
- 雙重解析同一輸出的機制
- 翻譯志願者外掛程式
- 訊息快取重送邏輯
