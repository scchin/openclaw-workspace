## 一、核心發現

### 技能代碼本身無問題

隔離測試（直接執行 `python3 run.py`）結果：
- ✅ **無翻譯痕跡**：無「最初是內用客戶」之類的翻譯文句
- ✅ **無雙重輸出**：仙草評論正常，Ivy League 合理出現 2 次（2則價格評論）
- ✅ **_wrap_chars 修復生效**：長評論不再被截成 16 行惨状

**結論：技能代碼（`run.py` / `query.py`）本身完全正確。**

## 二、真正的問題：OpenClaw 系統層行為

### 2.1 症狀對照

| 用戶描述的症狀 | 技能代碼測試結果 | 差異分析 |
|---------------|-----------------|---------|
| 🚗 開車 + 🚗 駕車 同時出現 | 技能只有「開車」，無「駕車」 | 系統層轉換 |
| 👤 仙草：原文 + 👤 仙草：翻譯 | 技能只有「初次內用」，無翻譯 | 系統層翻譯 |
| 評論一字排開（16行） | 技能最多2-3行 | 系統層截斷 |

### 2.2 OpenClaw WebChat 可能有的行為

根據症狀分析，OpenClaw 的 `webchat` channel 可能具備以下功能：

1. **自動翻譯志願者**
   - 對非正體中文內容自動翻譯
   - 翻譯後的內容與原文同時呈現

2. **雙重渲染機制**
   - 一次渲染原文，一次渲染翻譯
   - 導致「原文 + 翻譯」同時出現

3. **Markdown 增強渲染**
   - 對評論內容自動格式化
   - 對英文內容自動斷行

## 四、解決提案

### 方案 A：在技能層完全杜絕翻譯（推薦）

**方針：所有輸出直接使用 Google Maps 原文，不翻譯、不轉換**

修改 `format_full_output()` 中的所有評論呈現：
- 原文是簡體中文 → 保持簡體
- 原文是英文 → 保持英文
- 原文是正體中文 → 保持正體

不提供任何可以被「翻譯志願者」識別為需要翻譯的內容。

**實施方式：**
在所有輸出前加上語言標記，讓系統知道這些內容已處理過：
```
[zh-TW]初次內用，一直很喜歡鮪魚系列...
[en]Good food at a good price...
```

### 方案 B：在 OpenClaw 層停用翻譯功能

需要在 OpenClaw 的 `webchat` channel 設定中停用自動翻譯：
```json
{
  "channels": {
    "webchat": {
      "auto_translate": false,
      "preserve_original": true
    }
  }
}
```

### 方案 C：消除雙重渲染的來源

若問題是「一次渲染原文、一次渲染翻譯」導致，需要：
1. 在輸出中標記「此為翻譯版本」
2. 系統識別後不再重複渲染原文

## 五、根本原因結論

| 層面 | 評估 |
|------|------|
| **技能代碼** | ✅ 無問題，已多次驗證 |
| **Google Maps API 回傳** | ✅ 只有 `original_text`，無翻譯版本 |
| **deep_translator 殘留** | ✅ 已移除，代碼乾淨 |
| **OpenClaw 系統層** | ❌ 最可能的原因 |

**最可能的原因：OpenClaw webchat channel 具有自動翻譯功能，對技能輸出（非正體中文/英文內容）自動翻譯後，與原文同時呈現，形成「雙重輸出」。**
